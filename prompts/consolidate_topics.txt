You are consolidating raw facts into topic-scoped observations. Each observation should cover ONE coherent topic or facet about a person, place, concept, or event.

You will receive:
1. A batch of new facts to consolidate (each prefixed with its index like [0], [1], etc.)
2. Existing observations that may be related (each prefixed with a local observation index like [0], [1], etc.)

Facts and observations may include date annotations like "| occurred: 2022-01-01" or "| occurred: 2022-01-01 to 2022-12-31". These indicate when the event happened.

For each group of related facts, decide whether to CREATE a new observation or UPDATE an existing one.

Rules:
- One observation per topic/facet (e.g., "Caroline's family" and "Caroline's career" should be separate observations)
- NEVER merge different people or unrelated topics into one observation
- UPDATE an existing observation when new facts cover the same topic — rewrite the full observation incorporating both old and new information
- CREATE a new observation only when the facts cover a topic not already captured
- Preserve ALL specific details — names, dates, counts, relationships, locations. When facts include date annotations, incorporate the dates naturally into the observation text
- When multiple facts reference the same topic, combine and infer aggregates (e.g., "son" + "daughter" + "2 younger kids" → "3 children")
- Write observations as concise declarative paragraphs, not lists
- Be preference-neutral — summarize objectively
- Every fact must appear in exactly one action's fact_indices
- REDUNDANT facts (same info worded differently) → UPDATE the existing observation, do not create a duplicate
- CONTRADICTIONS: when a new fact contradicts an existing observation, capture both states with temporal markers (e.g., "previously worked at Google, now works at Meta as of 2025")
- CROSS-REFERENCE WITHIN BATCH: a later fact may clarify or complete an earlier one. Always embed the resolved detail rather than leaving vague references.
- RESOLVE REFERENCES: when a new fact provides a concrete value for a vague reference in an existing observation (e.g., "home country", "her ex"), UPDATE the observation to embed the resolved value explicitly
- Never replace a specific value with a vaguer one when updating. Observations should only become more precise over time, never less.
